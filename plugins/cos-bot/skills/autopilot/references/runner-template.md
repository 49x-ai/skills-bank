# Runner template

The skill writes this body verbatim to `~/.claude/commands/<slug>.md`
where `<slug>` = `<task>-autopilot`. The runner is invoked by the
sleeper: `claude --permission-mode bypassPermissions -p "/<slug>"`.

Each invocation: runs the underlying recipe, posts the result to
Telegram, evaluates the stop condition, and schedules its own next
firing (or halts).

---

## Placeholders

| Token | Replaced with |
|---|---|
| `{{SLUG}}` | `<task>-autopilot` (without the leading slash) |
| `{{TASK}}` | `<task>` (the underlying recipe slug) |
| `{{TASK_PATH}}` | absolute path to the recipe `.md` file (project or user scope) |
| `{{CHAT_ID}}` | Telegram chat-id from `access.json` |
| `{{MECHANISM}}` | `claude-p` or `schedule-wakeup` |
| `{{CADENCE_BLOCK}}` | inline JSON: shape + intervalOrTime + stop |
| `{{REGISTRY_PATH}}` | `~/.claude/channels/telegram/.cos-bot-autopilot.json` |
| `{{LOCKFILE_PATH}}` | `~/.claude/channels/telegram/.cos-bot-autopilot-{{SLUG}}.pid` |

## Template body

(Everything below the `---BEGIN---` line up to `---END---` is the
written runner content, with placeholders substituted.)

```
---BEGIN---
---
name: {{SLUG}}
description: Autopilot runner for /{{TASK}}. Fires the recipe, posts result to Telegram, schedules next run. Generated by /cos-bot:autopilot — do not edit by hand; re-arm to change cadence.
allowed-tools:
  - Read
  - Write
  - Bash(jq *)
  - Bash(date *)
  - Bash(ps *)
  - Bash(kill *)
  - Bash(nohup *)
  - Bash(printf *)
  - Bash(cat *)
  - Bash(test *)
  - Bash(mkdir -p *)
  - Bash(claude --permission-mode bypassPermissions -p)
---

# /{{SLUG}} — autopilot for /{{TASK}}

Generated runner. The cadence and stop policy are encoded below;
re-arm via `/cos-bot:autopilot /{{TASK}} ...` to change them.

Cadence:
{{CADENCE_BLOCK}}

Mechanism: {{MECHANISM}}
Registry:  {{REGISTRY_PATH}}
Lockfile:  {{LOCKFILE_PATH}}

---

## Execution

1. **Read the registry row** for `{{SLUG}}` via `jq`. If absent or
   `last_status != "active"`, exit silently (the autopilot was stopped
   while this fire was sleeping). Clean up the lockfile if it still
   points at our PID.

2. **Evaluate stop conditions** in order:
   - `now > stop_by`? → halt.
   - `len(runs) >= max_runs`? → halt.
   - prior run emitted `AUTOPILOT_STOP=<reason>`? → halt.
   On halt: set `last_status = "completed"`, post
   `"Autopilot completed: /{{TASK}} — <reason>"` to Telegram chat
   `{{CHAT_ID}}`, remove the lockfile, exit.

3. **Fire the recipe.** Spawn a nested `claude -p`:
   ```
   printf 'Run /{{TASK}} and post the result to my Telegram chat-id {{CHAT_ID}}.\nIf you want me to halt this autopilot, end your output with the line AUTOPILOT_STOP=<reason>.\nIf you want to change the cadence for the next run, end your output with the line AUTOPILOT_NEXT_DELTA_SECONDS=<int>.\n' \
     | claude --permission-mode bypassPermissions -p
   ```
   Capture stdout into `$OUTPUT`. Don't surface `$OUTPUT` to the
   user — the recipe already posts to Telegram.

4. **Parse signals** out of `$OUTPUT`:
   - `AUTOPILOT_STOP=<reason>` → set the halt flag for next-run-check.
   - `AUTOPILOT_NEXT_DELTA_SECONDS=<int>` → store in registry as
     `cadence.next_delta_seconds`.

5. **Update the registry.** Append `{ ts: <ISO>, status:
   ok|failed, signals: { stop?, delta? } }` to the row's `runs` array.
   If the recipe failed (non-zero exit or no output), increment
   `consecutive_failures` and apply backoff per
   `references/cadence.md` § *Failure backoff*. On success, reset
   `consecutive_failures = 0`.

6. **Compute next-fire-time.** Per `references/cadence.md` §
   *Absolute next-fire-time*. For adaptive: use
   `cadence.next_delta_seconds` if set, else the initial interval. For
   time-of-day: next occurrence of HH:MM on a matching day. For
   one-shot: halt after this fire.

7. **Re-arm** based on mechanism:

   - **`{{MECHANISM}} == "claude-p"`:**
     ```
     N=$(( <next_fire_ts> - $(date +%s) ))
     nohup sh -c "sleep $N && claude --permission-mode bypassPermissions -p '/{{SLUG}}' >/dev/null 2>&1" >/dev/null 2>&1 &
     SLEEPER_PID=$!
     printf '%s' "$SLEEPER_PID" > {{LOCKFILE_PATH}}
     # update registry: sleeper_pid = SLEEPER_PID, next_fire = <ISO>
     ```
   - **`{{MECHANISM}} == "schedule-wakeup"`:**
     Call `ScheduleWakeup` with `delaySeconds = N`, `prompt =
     "/{{SLUG}}"`, `reason = "autopilot {{SLUG}} next fire"`. The
     runner does not write a new sleeper PID — the live-session
     mechanism is self-tracking.

8. **Post status to Telegram** (only on halt or supervisor-detected
   re-arm; not on every successful fire — the recipe itself already
   posted its output).

---

## Exit codes

- `0` — fire succeeded (or stop condition met cleanly).
- non-zero — the runner itself errored (registry unreadable, lockfile
  conflict, etc). The supervisor's health-check will detect a stuck
  autopilot and re-arm.

---

## Idempotency / safety

- The runner re-reads the registry on every fire; in-flight cadence
  changes (e.g. `stop` while a fire is sleeping) are picked up at the
  start of the next firing.
- If two sleepers ever fire concurrently (shouldn't happen, but
  belt-and-suspenders), the second one detects a fresher
  `last_fire_at` and exits without re-firing.
- Recipe output is **not** echoed to stdout — only to Telegram. Keeps
  the runner's footprint small and prevents accidental log leaks.
---END---
```

## Notes for the skill author

- The "## Execution" block above is **prose-style** intentionally —
  it's a slash command that runs in a Claude session, not a shell
  script. The model handles the steps; the model gets the `Bash`
  calls right based on the embedded examples.
- The `allowed-tools` list is the **minimum** the runner needs. Don't
  expand without a specific reason — over-broad permissions make hook
  detection of denial less useful.
- `Mechanism: {{MECHANISM}}` in the YAML/markdown header is for human
  inspection; the actual runtime branch is in the Execution block.
- `{{CADENCE_BLOCK}}` is rendered as a fenced JSON block, e.g.:
  ```json
  {
    "shape": "time",
    "times": ["09:00"],
    "days": ["Mon","Tue","Wed","Thu","Fri"],
    "stop": { "stop_by": "2026-06-13T00:00:00Z", "max_runs": null }
  }
  ```
